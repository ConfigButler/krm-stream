# KRM resource stream protocol — the wire contract

> **Status:** specification, **v1 candidate**. The thin, language-neutral contract shared by two
> independently useful components: the **[resource-stream-gateway](../gateway/README.md)** (Go —
> *produces* this stream from a Kubernetes watch) and the **[live KRM reconcile engine](../docs/client-state-model.md)**
> (JS — *consumes* it to merge edits and render live status). Specified once, here, so the two sides
> cannot drift.
>
> **It streams KRM, and says so.** The payload is a Kubernetes object — `apiVersion`, `kind`,
> `metadata`, and whatever else that kind really has — not an abstracted "document". That fidelity is
> the point: the whole value of this family is being a faithful, live window onto the Kubernetes
> Resource Model. **Live `status` in a browser, with real RBAC and real attribution, is the product.**

A resource stream is a one-way feed, over Server-Sent Events, of the current-and-then-live state of a
**scope** (one named object, or a collection of a kind in a namespace). The server absorbs every
Kubernetes watch mechanic — `resourceVersion` arithmetic, bookmarks, relists, `410 Gone` — and emits a
small, stable vocabulary.

**Conventions.** MUST / MUST NOT / SHOULD / MAY are normative. Sections marked *(non-normative)* are
rationale and may not be relied on by an implementation. §10 is the complete conformance definition.

## 0. Versioning

- The protocol version is carried **in the endpoint path**: `…/resource-stream/v1`. There is no
  in-band negotiation in v1.
- The response MUST set `Content-Type: text/event-stream` and SHOULD set
  `X-KRM-Stream-Protocol: 1` for diagnosis.
- **Unknown event types MUST be ignored** by a consumer, not treated as errors: a minor addition to
  v1 (a new optional event type, a new optional envelope field) must not break an older client. A
  breaking change gets a new path segment (`/v2`).
- Unknown *fields* in a known event MUST be ignored likewise.

## 1. Identity: `target` + `uid`

- Every object is keyed on **`metadata.uid`**. A name can be deleted and recreated; the `uid` cannot
  be reused. Consumers MUST key on `uid` and MUST NOT key on `name`.
- A `uid` is unique **within one upstream target** (one API server / kcp workspace / cluster), not
  globally. A consumer holding objects from more than one stream MUST key on **`(target, uid)`**, or
  keep one map per stream. The `target` is delivered on every `reset` (§4).
- The SSE `id:` field is **not used in v1** (§7). "Object identity" is always `metadata.uid`; if an
  envelope field is needed for it in a future version it will be named `resourceUid`, never `id`.

## 2. The payload: a complete projected KRM object

An event's `object` is **the complete unstructured JSON representation of the resource as served by
Kubernetes, minus exactly the fields removed or redacted by the stream's declared projection (§3).**

It is not a schema. It is not "`metadata` + `spec` + `status`". A `ConfigMap` has `data` and no
`spec`; a `Secret` has `type` and `stringData`; a CRD may define any field at the object root; a
resource may have no `status` at all. The protocol carries **whatever is there**, verbatim.

```jsonc
{
  "apiVersion": "apps/v1",
  "kind":       "Deployment",
  "metadata": {
    "uid":             "9f1c…",   // identity — consumers key on it (per target)
    "name":            "web",
    "namespace":       "shop",
    "resourceVersion": "12345",   // opaque to consumers (§6)
    "labels":          { "app.kubernetes.io/name": "web" },   // keys carry dots AND slashes
    "annotations":     { "note": "keep" },
    "creationTimestamp": "…", "generation": 4, "ownerReferences": [ … ]
  },
  "spec":   { /* free-form nested JSON */ },
  "status": { /* free-form nested JSON */ }
  // …and any other root field the kind actually defines
}
```

Normative rules:

- The gateway MUST NOT flatten, reorder, coerce, or schema-check any part of the object. Nested JSON
  is carried through as-is. A gateway that "helpfully" normalizes silently corrupts CRDs it has never
  heard of.
- The gateway MUST NOT synthesize fields that the server did not send.
- **When the resource has a `status`, the editing projection MUST NOT omit it merely because it is
  read-only.** A status-only view and an editing view are the *same stream*; the consumer decides
  which regions are editable. (A security projection MAY still redact specific paths — §3.)
- The gateway MUST NOT forward Kubernetes control frames — `BOOKMARK`, watch `ERROR`, or
  metadata-only / partial objects — as `added` or `modified`. A partial object delivered as an update
  would blank a consumer's state.

*(Non-normative.)* Field conventions worth knowing, not an exhaustive object schema: `spec` is desired
state and is usually the editable region; `status` is controller-owned and usually read-only;
`ConfigMap.data` / `binaryData` and `Secret.data` / `stringData` / `type` carry editable payload at the
object **root**, outside `spec` — a consumer must not assume every editable region lives under `spec`.

## 3. Projection and redaction — part of the wire, not an implementation detail

An object on the wire is complete **within a projection**. The projection is named and machine-readable
so a consumer can always tell these three apart:

1. a field that does not exist upstream;
2. a field the gateway **removed** — never shown, and never named;
3. a field that **exists upstream but whose value was withheld** — the path is named in
   `redacted[].path`, and the value is **not on the wire at all**.

Rules:

- Every `reset` MUST carry a **`projection`** identifier (e.g. `"krm-full/v1"`). Its removal rules
  are declared by the gateway and are stable for the life of that identifier.
- Every `added` / `modified` MUST carry **`redacted`**: an array of `{path, rev}` records. `path` is
  an RFC 6901 pointer and `rev` is a positive revision that increments when the withheld value changes
  on this connection. Empty array when nothing is redacted. It MUST be present, not merely optional.
- **A redacted value MUST NOT appear on the wire — not even as a placeholder.** The gateway omits it;
  `redacted[].path` names it. This is normative, and §3.1 says why.
- **A container the projection empties MUST be removed with its contents, not left behind as `{}`.**
  Redact every value in a `Secret`'s `data` and the object carries **no `data` key at all** — not
  `data: {}`. An empty map that exists *only because the gateway emptied it* is the gateway's artifact,
  not the server's state: `data: {}` asserts *"this Secret has an empty data map"*, which is a
  **different fact** from *"this Secret's data is not yours to see"* — and the second is what happened.
  `redacted` is what says so, and it is the only thing that does. (The same rule already applies
  to `metadata.annotations` when the projection removes the last one. A container the *server* sent
  empty is kept: the gateway removes what it removed, and nothing else.)
- A consumer MUST NOT treat the absence of a path in `object` as a deletion of that path if the path
  appears in `redacted` or is removed by the projection.
- **The write path MUST NOT round-trip a projection.** A consumer MUST NOT send a whole-object `PUT`
  built from a projected object — it would delete every path the projection removed. Saves are a
  constrained patch (RFC 7386 merge patch or server-side apply) over the paths the consumer actually
  edited.

### 3.1 Why a redacted value is *omitted* and not *masked* — normative

The obvious design is to substitute a placeholder (`"**REDACTED**"`, `"••••••"`) for a Secret's value.
**Do not.** It is the single most dangerous thing a projection can do, and it is dangerous precisely
*because* it looks harmless:

> A browser holding `{"token": "**REDACTED**"}` can **save it back**. A merge patch carrying that
> value writes the literal string `**REDACTED**` **over the real Secret**. The token is destroyed —
> by an ordinary save, that looked from the browser like a green tick.

The placeholder is the only poisoned value in the system, and a gateway that emits one *creates* that
hazard for every consumer downstream. Omitting the value removes it at the source: **there is nothing
to write back.** A hazard that cannot arise needs no guard, and a guard is what the alternative
requires — on every save endpoint, forever, correctly.

Nothing is lost. `redacted` is authoritative and mandatory, so it carries everything the mask
carried: the consumer still learns that `token` **exists** and is withheld, and a UI renders `••••••`
from *that*. **The mask is something a consumer draws. It is not something the wire carries.**

> **This protocol specifies a READ path.** There is no `apply`, no `PATCH` and no save endpoint in it.
> A change reaches the cluster the way it always did: through the Kubernetes API, from **your** server.
> Your save endpoint SHOULD still refuse a patch touching a path the projection *removed* (e.g.
> `managedFields`) — the consumer never saw those either. But the catastrophic case, the one that
> destroys a Secret, is closed here, in the projection, where it was created.

Every gateway MUST define at least these two projections and say which is in force:

| projection | removes |
|---|---|
| `krm-raw/v1` | `metadata.managedFields`, `metadata.annotations["kubectl.kubernetes.io/last-applied-configuration"]` — nothing else |
| `krm-full/v1` | the above, **plus** keys-only disclosure for `Secret` — every value under `data`/`stringData` is **deleted** (not masked) and its path named in `redacted` (§3.1) |
| `krm-spec/v1` | `krm-full/v1`, plus `status` is omitted and status-only changes are suppressed |

There is no unspecified "optionally other server-side bookkeeping". If a gateway removes a field, that
removal is part of a named projection or it does not happen.

*(Non-normative.)* Removing `managedFields` and `last-applied-configuration` exists so a human editor
is never shown, and never round-trips, machinery — not for byte savings.

## 4. The event vocabulary

Each SSE `data:` frame carries one JSON object `{ "seq": N, "type": …, … }`. `seq` starts at one per
connection and is gap-free for emitted events. Heartbeats are SSE comments
(`: heartbeat`), not events, and are a no-op to consumers.

| type | fields | meaning | consumer action |
|---|---|---|---|
| `reset` | `seq`, `target`, `scope`, `projection` | a snapshot cycle begins **now** | mark every known uid "unseen"; **do not prune yet** |
| `added` | `seq`, `object`, `redacted` | an object is in scope | **replace** the authoritative object for its uid; mark "seen" |
| `modified` | `seq`, `object`, `redacted` | the object changed (any field) | **replace** the authoritative object for its uid; mark "seen" |
| `deleted` | `identity`, `object?` | the object left scope | drop that uid |
| `synced` | — | the snapshot cycle is complete | prune every uid still "unseen" |
| `error` | `code`, `message`, `terminal`, `retryAfterMs?` | backend trouble | surface; if `terminal`, close the connection |

`added` and `modified` are the **only two** upsert spellings. There is no `upsert` alias and no
"configured spelling" — two independently implemented components need one deterministic vocabulary.
Both mean *"here is this object's current complete state"*, and a consumer MUST treat them
identically. They are kept distinct because they are verbatim KRM (they are what the watch says) and
because a UI legitimately wants to animate an arrival differently from an update.

### 4.1 `added` / `modified` — replacement, not merge

> A consumer maintains the latest authoritative server representation by **replacing** the complete
> projected object for that uid. An *editing* consumer MAY separately reconcile that representation
> against local edits.

This is the single most important consumer rule, and the previous draft of this document got it wrong.
A generic deep-merge of complete server objects is **incorrect**: it resurrects ghosts.

```jsonc
old: { "spec": { "a": 1, "b": 2 } }
new: { "spec": { "a": 1 } }          // server removed b
// deep-merge → { "a": 1, "b": 2 }   ✗ b is a ghost
// replace    → { "a": 1 }           ✓
```

Two distinct concerns, and only the first is a protocol conformance requirement:

1. **Stream reducer (protocol):** replace the complete authoritative server object for that uid.
2. **Live reconcile engine (not the protocol):** recursively three-way reconcile the *previous* server
   object (base), the locally edited draft (ours), and the *new* complete server object (theirs). See
   the [client state model](../docs/client-state-model.md). The gateway neither knows nor cares.

### 4.2 `deleted` — a tombstone with identity

`deleted` is the one event that carries no complete object, because there may not be one. It carries
an **identity**:

```jsonc
{
  "type": "deleted",
  "identity": {
    "uid":        "9f1c…",       // REQUIRED
    "apiVersion": "apps/v1",     // REQUIRED
    "kind":       "Deployment",  // REQUIRED
    "namespace":  "shop",        // REQUIRED when the resource is namespaced
    "name":       "web"          // REQUIRED
  },
  "object": { /* OPTIONAL: the final projected state, when the gateway has it */ }
}
```

- `uid` is what the consumer acts on. The rest exists for logs, UI messages ("`shop/web` was
  deleted"), and defensive validation.
- If the gateway **cannot** recover a trustworthy `uid` (e.g. a degenerate informer deletion
  tombstone), it MUST NOT emit an ambiguous `deleted`. It MUST begin a new snapshot cycle (§5)
  instead, and let `reset`…`synced` pruning remove the object.

### 4.3 `error`

```jsonc
{ "type": "error", "code": "FORBIDDEN", "message": "Access to this scope was revoked",
  "terminal": true, "retryAfterMs": null }
```

`code` MUST be one of a stable, machine-readable set — a free-form `message` alone is not enough for
stable client behavior. v1 codes:

| code | terminal | meaning |
|---|---|---|
| `FORBIDDEN` | yes | the caller lost (or never had) access to this scope |
| `UNAUTHENTICATED` | yes | the caller's credential expired or was rejected |
| `SCOPE_INVALID` | yes | the requested scope is not allowlisted / not resolvable |
| `UPSTREAM_UNAVAILABLE` | no | the gateway cannot reach the API server right now |
| `RESYNC_REQUIRED` | no | continuity was lost; a new snapshot cycle follows |
| `SLOW_CONSUMER` | yes | this subscriber fell too far behind and was dropped |
| `INTERNAL` | either | anything else; `terminal` says whether to give up |

- `terminal: true` MUST be the **last** event on the connection, and the gateway MUST then close it.
- A consumer using native `EventSource` MUST call `close()` on a terminal error — `EventSource`
  reconnects automatically otherwise and will hammer a forbidden scope forever.
- `retryAfterMs` is advice for a non-terminal error; `null`/absent means "use your own backoff".

## 5. Snapshot cycles — `reset` … `synced`

A **snapshot cycle** is `reset`, zero or more `added`, then `synced`. Between `synced` and the next
`reset`, events are live deltas.

```text
reset
added …            (one per object currently in scope — zero or more)
synced
modified … / added … / deleted …   (live)
```

Normative:

- **Every scope uses snapshot framing — including a single named object.** There is no exemption. A
  named object that does not currently exist is:

  ```text
  reset
  synced
  ```

  This is exactly the case a "named scopes may skip the snapshot" rule gets wrong: if the object is
  deleted while the consumer is disconnected, the reconnect's fresh lookup finds nothing, and without
  a `reset`…`synced` boundary the consumer keeps the old uid as a **ghost forever**. One framing, one
  consumer state machine, both scope types.

- **A stream MAY contain many snapshot cycles.** A cycle is not tied to a physical connection. The
  gateway MUST begin a new cycle whenever it loses upstream continuity — an expired `resourceVersion`
  (`410 Gone`), an informer relist, a cache reset, a lost replay buffer, an authorization-scope
  refresh — even if the SSE connection itself is perfectly healthy. It SHOULD precede such a cycle
  with a non-terminal `error` of code `RESYNC_REQUIRED`. This is legal at any point:

  ```text
  modified  modified  reset  added  added  synced  modified
  ```

- **Pruning is gated on `synced`.** On `reset` a consumer marks all known uids unseen; on each `added`
  it marks that uid seen; on `synced` it prunes every still-unseen uid. A consumer MUST NOT prune
  before `synced`. If the connection drops mid-cycle, the incomplete snapshot MUST NOT cause any
  pruning — the state from *before* the `reset` is retained until a cycle completes.

- **Gap-free handoff.** The gateway MUST guarantee that every state change occurring after a snapshot's
  logical boundary is either delivered as a subsequent event **or** causes a new snapshot cycle.
  *How* it achieves this is the gateway's business and is deliberately absent from this contract.
  (Streaming lists, list-then-watch at a pinned `resourceVersion`, and an informer cache with a replay
  buffer are all conforming backends — see the [gateway guide](../gateway/README.md).)

- A snapshot generation identifier is unnecessary: SSE preserves ordering within a connection, so
  `reset` and `synced` are self-framing.

## 6. Ordering, delivery & the state guarantee

The service is **state-convergent, not event-complete.** It does not promise to deliver every
intermediate Kubernetes transition; it promises that what you end up holding is right.

**The primary invariant:**

> For each logical stream: immediately after `synced`, and after each subsequent `added` / `modified` /
> `deleted` until the next `reset`, the consumer's authoritative object map equals the gateway's
> projected view of server state at the corresponding logical stream position. Any loss of continuity
> produces another snapshot cycle rather than silent continuation.

Between `reset` and `synced` the consumer deliberately holds transitional state, and there is always
transport latency; neither is a violation.

What is guaranteed:

- **No persistent ghosts.** A deletion the consumer missed is resolved by the next snapshot cycle.
- **No silently missed final state.** The last state of an object is always delivered or snapshotted.
- **Ordered application within one stream.** SSE preserves order; there is no cross-stream ordering.
- **Per-object monotonicity.** Within one snapshot cycle, the gateway MUST NOT emit a state for a uid
  that is *older* than one it already emitted for that uid. This is what makes coalescing safe (below)
  and is the one guarantee a naive "just relay the watch" gateway does not give you.
- **At-least-once, idempotent.** A reconnect replays a snapshot, so a consumer may see the same object
  version twice. Applying an event MUST be idempotent.

What is **not** guaranteed:

- **Every intermediate version.** The gateway MAY coalesce rapid updates to the same uid and emit only
  the latest. Controllers rewrite `status` continuously; a browser that renders at 60Hz has no use for
  200 intermediate `observedGeneration` bumps. Per-object monotonicity makes this safe: a coalesced
  stream converges to the same final state.

**`resourceVersion` is opaque to consumers.** A consumer MAY echo `metadata.resourceVersion` back on
save for optimistic concurrency, and MUST NOT parse, compare, or order by it.

> *(Non-normative — and the reason this protocol can promise per-object monotonicity at all.)* Within a
> single upstream target, a modern Kubernetes API server's `resourceVersion` for a given **resource
> type** is orderable as a monotonically increasing decimal (it is the backing store's revision). The
> **gateway** relies on this internally — to coalesce safely, to drop a stale replay after a relist,
> and to assert it never hands a consumer a state older than one already sent. It is deliberately
> **not** exposed as an ordering primitive on the wire: it is not comparable across targets or across
> aggregated API servers, and a browser has no business doing revision arithmetic. See the gateway
> spec §3.
>
> **Three things an implementer must get right here, and we got the first two wrong** (see
> [docs/facts/kubernetes-api-concepts.md](../docs/facts/kubernetes-api-concepts.md)):
>
> - A `resourceVersion` is an **arbitrary-bitsize** decimal. Kubernetes' own documented example is 40
>   digits long, so it does **not** fit in an int64, and parsing it into one silently fails — which
>   makes a staleness check drop live updates. Compare as strings: **longer is greater; equal lengths
>   compare lexicographically.** (A plain lexicographic compare is also wrong — it calls `"9" > "10"`.)
> - Orderability is only defined **within one resource type** (same group + resource). Two Deployments
>   compare; a Pod and a Deployment do not.
> - **From Kubernetes 1.35, orderability is a Certified Kubernetes conformance requirement**, for
>   built-in types *and* custom resources. A gateway targeting 1.35+ may therefore **rely** on it — and
>   should say so out loud when an upstream breaks it, rather than degrading in silence. The one case
>   that remains genuinely unordered is an **aggregated / extension API server**, which that conformance
>   test does not cover; a gateway serving one needs an explicit opt-out, and must then drop nothing it
>   cannot order (a duplicate is harmless — §6 requires idempotent apply — while a wrong drop is data
>   loss). See `gateway.ResourceVersionOrdering`.

## 7. Transport & authentication

- **SSE.** `text/event-stream`, one JSON object per `data:` frame, `: heartbeat` comment every ~20s so
  intermediaries don't idle the connection out.
- **SSE `id:` fields are deliberately unused in v1.** Every new physical connection begins a complete
  snapshot cycle, so there is no delta replay and no `Last-Event-ID` resume. A gateway MUST NOT emit
  SSE `id:` lines in v1 — in particular it MUST NOT put a resource `uid` there, which would give
  `Last-Event-ID` an incorrect meaning on the browser's automatic reconnect.
- **Authentication.** Native `EventSource` cannot send custom request headers, so a v1 gateway MUST
  support **same-origin session cookies** (`SameSite`, `Secure`, `HttpOnly`) as its baseline
  authentication for the stream. A gateway MAY additionally accept a `Authorization: Bearer` header,
  in which case the consumer needs a **fetch-based SSE reader** rather than native `EventSource`; the
  JS reference consumer provides both.
- A gateway MAY accept a **short-lived, single-scope stream ticket** in the URL for cross-origin use.
  **Long-lived bearer credentials MUST NOT be placed in the URL** — URLs land in proxy logs, browser
  history and `Referer` headers.

## 8. Scope & the request

A stream is opened for a **normalized, server-validated scope**, never a raw Kubernetes URL. The
browser asks for a *logical* scope; the server maps it to an allowlisted target + GVR + the caller's
permitted namespace, and denies by default. The canonical scope is:

```text
target, group, version, resource, [namespace], [name], [labelSelector]
```

- **`target`** — an **allowlisted identifier** for one upstream API (a cluster, a kcp workspace, a
  tenant). Required as soon as a gateway can reach more than one. The browser MAY select from
  allowlisted target ids; it MUST NOT supply an API-server URL. It is echoed on every `reset` and is
  the first half of the consumer's `(target, uid)` key (§1).
- **`namespace`** — present for one namespace. When absent, the gateway host must distinguish a
  cluster-scoped resource from an all-namespaces watch and explicitly authorize the latter.
- **All-namespace collection watches require explicit policy.** They are never implied by a missing
  namespace; hosts should keep them rare because snapshot and fan-out cost scale with every namespace.
- **`name`** — a single-object scope. Subject to the same snapshot framing as a collection (§5).
- **`labelSelector`** — OPTIONAL, and **gateway-validated**: a gateway MAY accept one and MUST be free
  to reject it. It participates in the normalized scope key (so two callers asking for the same
  selector share one upstream watch).
- **Field selectors are not supported in v1.**

The scope is fixed for the life of the stream. It is not repeated in per-object payloads; it is
delivered once per cycle on `reset`.

### 8.1 The encoding — normative

v1.0 named these fields and never said how they reach the server. That omission had a cost: each end
invented its own spelling and nothing compared them. The encoding is therefore pinned here, and the
shared corpus (`conformance/scopes.yaml`) is what holds both ends to it — the client builds the
query, the gateway parses that exact query back.

A scope is carried as **URL query parameters** on the stream request, one per field, spelled exactly
as above (`target`, `group`, `version`, `resource`, `namespace`, `name`, `labelSelector`). A client
MUST emit them in that order; a gateway MUST accept them in any order.

```text
GET /resource-stream/v1?group=apps&version=v1&resource=deployments&namespace=app
```

- `resource` is the **plural, lowercase API name** (`configmaps`), never the Kind (`ConfigMap`).
- `resource` and `version` are **REQUIRED**. A gateway MUST NOT default either: a defaulted `version`
  streams objects of a shape the caller did not ask for, and a defaulted `resource` streams the wrong
  objects entirely.
- An omitted field is an **absent** field, never an empty one.
- Identifier fields are validated using their applicable Kubernetes/DNS naming rules. A gateway MUST
  reject an invalid identifier. When accepted, `labelSelector` MUST instead be parsed and validated
  using Kubernetes label-selector syntax.
- A gateway MAY tolerate additional query parameters of its own.
- **A gateway MUST REFUSE — not ignore — any parameter that supplies an API-server address, endpoint
  or credential** (`server`, `url`, `endpoint`, `kubeconfig`, `token`, …). Silently dropping such a
  parameter satisfies the letter of "the browser never supplies an API-server URL" while making an
  SSRF probe indistinguishable from an ordinary request. The refusal is what makes the rule
  observable.

Rejections are `SCOPE_INVALID` and **terminal** — a browser's `EventSource` reconnects on its own,
and would otherwise retry a malformed scope forever.

## 9. Notes (non-normative)

- **Most `modified` events are `status`-only.** Controllers write `status` constantly — conditions,
  `observedGeneration`, replica counts. On a typical workload this, not editing, is the dominant
  traffic on a live stream, and it is exactly what the live-status-watch use case renders. It is also
  why the gateway's fan-out, coalescing and backpressure rules matter far more than they first appear.
  This is an observation about workloads, not a protocol rule.
- **`added` ≡ `modified` for state purposes**, but they carry real information for a UI (arrival
  animation vs. change flash) and for diagnostics, which is why both are retained.
- **Why `metadata.uid`:** it is the only identifier Kubernetes never reuses. A delete-and-recreate
  under the same name produces a *different* object, and keying on `uid` makes the consumer notice.

## 10. Conformance

A **gateway** conforms iff:

1. It serves `/…/v1` and sets `Content-Type: text/event-stream`.
2. Every stream, and every recovery, is framed as `reset` … `synced` — **for named and collection
   scopes alike**, including the empty named case (`reset`, `synced`).
3. It begins a new snapshot cycle on any loss of upstream continuity, without requiring the SSE
   connection to drop, and guarantees a **gap-free handoff** from each snapshot to live events.
4. Every `added` / `modified` carries a **complete projected object** and a `redacted` array.
   Every `deleted` carries a complete `identity` with a trustworthy `uid` — or is not emitted, and a
   new snapshot cycle is begun instead.
5. It emits `reset.projection`, and its projections' removal rules are declared and stable.
6. It never forwards `BOOKMARK`, watch `ERROR`, or metadata-only/partial objects as `added`/`modified`.
7. It carries every other field of the object **verbatim** — nothing flattened, reordered, coerced,
   schema-checked or synthesized — except fields a named projection ignores.
8. It never emits, within a snapshot cycle, a state for a uid older than one already emitted.
9. It emits no SSE `id:` lines; it heartbeats; a `terminal` error is its last event and it then closes.
10. Its scope is allowlisted and server-normalized; it never accepts a raw API URL.

> There is deliberately no rule here about rejecting a save. A gateway has no write path (§3) — the
> save is the host's endpoint, and so is the duty to refuse a patch that touches a redacted path. A
> conformance rule the gateway cannot execute would be a promise the wire does not keep.

A **consumer** conforms iff:

1. It keys on `(target, uid)`, never on `name`.
2. It treats `added` and `modified` identically and **replaces** the authoritative object; any
   three-way merge with local edits is layered *above* that, never in place of it.
3. It marks unseen on `reset`, seen on `added`, and prunes only on `synced` — never on a partial cycle.
4. It tolerates repeated snapshot cycles on one connection, and reconnect replays (idempotent apply).
5. It never parses or orders by `resourceVersion`.
6. It never writes a redacted or projection-removed path back. (It cannot accidentally write a redacted VALUE: there is none — §3.1.) A mask it renders for a redacted path is its own, drawn from `redacted`.
7. It renders `binaryData` / non-UTF-8 values as opaque ("N bytes"), never as editable text.
8. It ignores heartbeats, and ignores unknown event types and unknown fields.
9. It calls `EventSource.close()` on a `terminal` error and reconnects for a fresh snapshot on a `seq` gap.

### 10.1 The conformance tests that actually matter

These are the eight that catch real bugs; the rest are variations.

| # | Scenario | Expected |
|---|---|---|
| 1 | delete a **collection** object while the consumer is disconnected | after reconnect's `reset`…`synced`, the uid is pruned — no ghost |
| 2 | delete a **named** object while the consumer is disconnected | reconnect yields `reset`, `synced` (empty) — the uid is pruned. *This is the test the "named scopes may skip the snapshot" rule fails.* |
| 3 | delete and recreate under the same **name**, new uid | consumer holds the new uid only; the old one is pruned; no state bleeds across |
| 4 | disconnect **between `reset` and `synced`** | nothing is pruned; pre-`reset` state survives until a cycle completes |
| 5 | lose **upstream** continuity without closing the SSE connection (relist / `410 Gone`) | a fresh `reset`…`synced` arrives mid-stream; state converges; no ghost |
| 6 | server removes a **nested** field (`spec.b`) | consumer's authoritative object no longer has it (replacement, not deep-merge) |
| 7 | redact a `Secret` value, then save an unrelated field | the value is not on the wire at all, so it cannot be written back; the unrelated edit still saves normally. Keys-only disclosure survives via `redacted` (§3.1) |
| 8 | run one randomized operation sequence through **two different backends** (streaming-list, informer fan-out) | both produce **identical** final projected consumer state |
